- 2026-07-23: Root-caused the recurring "claude-free stops and needs --continue" reports as an architectural problem, not a fixable edge case: once Kompass forwards live SSE bytes to the client, a provider dying mid-generation can't be un-sent, and the earlier fix (an in-stream Anthropic `error` event on abnormal EOF) assumed Claude Code auto-retries on it — repeated live reports proved it does not; it just stops. Total chain exhaustion also now cascades across lanes (FAST→SIMPLE→AGENTIC→HARD) before giving up, and even full exhaustion returns a synthetic 200 assistant notice, never a raw protocol error — so Claude Code always sees a normal completed turn and never needs a manual `--continue`. - 2026-07-24: M6 (context fit filter) design decisions. New optional YAML fields `default_ctx` (provider), `ctx`/`max_out`/`tpm` (per model_limits entry) — all optional/defaulted, verified by a CI regression test (test/fit.test.ts) that an unmodified v1 config (no ctx anywhere) routes byte-identically. Estimator is `bytes / ratio[provider]` (no tokeniser — SPEC_V2 §4 non-goal), ratio seeded 3.6, isolate-local Map self-calibrated via EWMA (α=0.2) from each response's actual `usage.input_tokens` — same "module-scope cache" pattern as privacy.ts's compiled guard, deliberately NOT stored in the Durable Object (M6 scope explicitly excludes `src/do/` internals; an isolate serves many requests so it still converges within a session). Fit filter runs in `routeRequest` (router.ts) strictly after `ctx.privacySensitive` is decided and before the DO's `filterChain` quota/cooldown pass, confirmed by a dedicated test that a bigger `trains_on_data:true` model is never preferred over a smaller safe one even though both fit. Unknown-ctx entries are kept and ranked after known-fitting entries (never hard-dropped); a config with zero ctx-bearing fields anywhere is provably a no-op reorder. Measured CPU cost of `filterChainByFit` on a 400KB body / 10-entry chain: ~0.005ms per call (test/fit.test.ts, well under the 1ms budget) — cheap because the filter only ever touches the byte _length_ already captured once at ingress, never the body content itself. LONGCTX's heuristic threshold (dispatcher.ts) is now `smallestCtx(AGENTIC chain)` with the old `60_000` as the no-data fallback; with the real M6 provider data below, that's now 128,000 (mistral/devstral-medium-latest is AGENTIC's smallest declared window) — deliberately not hardcoded, so it moves automatically as lane composition changes. The "request fits nothing anywhere" edge case (SPEC_V2 §6) gets its own synthetic notice (`noFitNotice` in escalation.ts) distinct from `HARD_EXHAUSTED_NOTICE`, naming the largest ctx actually configured across every lane that was tried — only fires when literally every attempt across every escalated lane was `skipped-too-large` (any other kind of skip/failure falls back to the generic exhausted notice).
